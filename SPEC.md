# 仕様書 (SPEC.md)

ログインバリデーター（Login Validator）のシステム設計および詳細仕様。

## システムアーキテクチャ

### 構成
- **フロントエンド（UI）**: Next.js (App Router), Tailwind CSS
- **バックエンド（検証API）**: Node.js (pages/api/validate/route.ts)
- **接続・検証コア**: lib/connectionValidator.ts

### デュアル入力検証のデータ構造
1セットにつき「URL」「ユーザーID」「パスワード」を入力可能とし、以下の2系統を同時に検証できる：
1.  **WordPressのログイン情報**: 管理者権限（Administrator）までスキャン。
2.  **レンタルサーバーの管理画面**: URLへの接続性（HTTP 200/302等）を主に確認。

---

## ログイン検証ロジック

### 1. WAF（ファイアウォール）回避アルゴリズム
いきなり `wp-login.php` を開くのではなく以下の 2 段階でアクセス：
- **Step 0**: サイトのトップ (`/`) に Aアクセスし、サーバーから正規のセッション（クッキー）を受け取る。
- **Step 1**: 取得したクッキーと、トップページを `Referer` として指定し、ログイン画面へアクセス（HTTP 403 エラーを回避）。

### 2. WordPress ログイン検証の流れ
1.  **URL自動探索**: 指定された URL でログインフォームが見つからない場合、自動的に `/wp-admin/` や `/wp-login.php` を付与して再試行。
2.  **ログイン試行**: 取得した `log`, `pwd`, `wp-submit`, `_wpnonce`, `testcookie` 等のパラメータをセットして POST 送信。
3.  **成功判定**:
    - HTTP 302 リダイレクト（`wp-admin/` への遷移）の発生を確認。
    - `wordpress_logged_in_...` クッキーが発行されているかを確認（ログイン成功の決定的な証拠）。
4.  **管理者権限（Privilege）チェック**:
    - ログイン成功直後に、クッキーを引き継いだ状態で `wp-admin/` ページを取得（GET）。
    - HTML内から `#menu-plugins`, `#menu-settings`, `#menu-users` などの管理者限定の ID を検出。
    - 管理者メニューがあれば「管理者権限を確認済み」として成功、なければ「権限不足」として警告。

## エラーハンドリング・フィードバック

### 判定結果コード
- **LOGIN_SUCCESS**: ログインに成功し、権限も確認済み。
- **AUTH_FAILED**: パスワードが違う、またはログインできても管理者権限がない。
- **LOGIN_PAGE_NOT_FOUND**: ログインフォームを特定できなかった（SiteGuard 等でURLが変更されており、かつ自動探索でも見つからなかった場合）。
- **URL_NOT_FOUND**: ドメイン自体にアクセスできない（DNSエラー、SSLエラー等）。

### クライアントへの再送依頼文
問題が発生した際、依頼人にそのままコピーして送れる「再送願いテンプレート」を自動生成。

## コピペ・サマリー
- **デフォルトパスワード表示**: 登録されたパスワードをサマリー上で伏せずに表示（共有の手間を省く）。
- **一括コピーテンプレート**: 
  - a. サイトのアドレス
  - b. サーバー管理画面ログイン情報
  - c. WordPressログイン情報
  - d. ヒント情報
  上記の内容を一つのブロックにまとめてクリップボードに保存可能。
